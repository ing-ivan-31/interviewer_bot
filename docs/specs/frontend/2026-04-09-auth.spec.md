# Auth — MSAL + Next.js App Router

**Status:** Draft
**Author:** @spec-writer
**Date:** 2026-04-09

## Purpose

Integrate Microsoft Authentication Library (MSAL) into the Next.js frontend to
authenticate users via Azure AD. All routes except `/login` are protected —
unauthenticated users are redirected to `/login`. MSAL manages the token lifecycle
(acquisition, caching, and silent refresh) internally.

---

## Environment Variables

Add to `apps/frontend/.env.local`:

```env
# Azure AD Application (client) ID — from your app registration
NEXT_PUBLIC_MSAL_CLIENT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

# Azure AD authority URL — e.g. https://login.microsoftonline.com/{tenant-id}
NEXT_PUBLIC_MSAL_AUTHORITY=https://login.microsoftonline.com/your-tenant-id

# Redirect URI registered in Azure AD (must match exactly)
NEXT_PUBLIC_MSAL_REDIRECT_URI=http://localhost:3000/auth/callback
```

## Auth Flow

```
1. User visits any protected route (e.g. /evaluation)
2. middleware.ts checks for MSAL account in session
3. No account found → 302 redirect to /login
4. User clicks "Sign in with Microsoft" on /login page
5. MSAL triggers loginRedirect() → browser redirects to Azure AD
6. User authenticates at Azure AD (which may federate to Okta)
7. Azure AD redirects to /auth/callback with authorization code
8. /auth/callback page calls handleRedirectPromise()
9. MSAL exchanges code for tokens, stores in sessionStorage cache
10. User redirected to original protected route
11. lib/auth/auth.ts:getAccessToken() acquires token silently from cache
12. API calls include: Authorization: Bearer <access_token>
```

## Props / Inputs

### `MsalProvider` (msal-provider.tsx)

| Name | Type | Required | Description |
|------|------|----------|-------------|
| children | ReactNode | yes | App content to wrap |
| instance | PublicClientApplication | yes | MSAL app instance from msal-config.ts |

### `getAccessToken()` (auth.ts)

| Name | Type | Required | Description |
|------|------|----------|-------------|
| scopes | string[] | no | OAuth scopes. Defaults to `["openid", "profile", "email"]` |

**Returns:** `Promise<string>` — the access token string

### `signOut()` (auth.ts)

No parameters. Calls `msalInstance.logoutRedirect()` and clears session.

## MSAL Configuration (msal-config.ts)

```typescript
// Key config values — not implementation, just what must be set:
{
  auth: {
    clientId: NEXT_PUBLIC_MSAL_CLIENT_ID,
    authority: NEXT_PUBLIC_MSAL_AUTHORITY,
    redirectUri: NEXT_PUBLIC_MSAL_REDIRECT_URI,
  },
  cache: {
    cacheLocation: "sessionStorage",   // survives page refresh within tab
    storeAuthStateInCookie: false,     // not needed for modern browsers
  }
}
```

## Business Rules

1. **All routes except `/login` and `/auth/callback` require authentication.**
   Unauthenticated requests are redirected to `/login` by `middleware.ts`.

2. **Token acquisition is always silent first.**
   `getAccessToken()` calls `acquireTokenSilent()`. If the silent call fails
   (expired refresh token), the user is redirected to `/login` for a fresh login.

3. **Post-login redirect.**
   After successful authentication at `/auth/callback`, the user is redirected to
   the route they originally tried to access. If no prior route, redirect to `/evaluation`.

4. **`MsalProvider` must be a Client Component.**
   Next.js App Router defaults to Server Components. `MsalProvider` requires browser
   APIs — it must be a Client Component (`"use client"`) wrapping the layout.

5. **Token is passed to API calls via Authorization header.**
   Every HTTP call to the backend includes `Authorization: Bearer <access_token>`
   obtained via `getAccessToken()`.

6. **Sign-out clears MSAL cache and redirects to `/login`.**
   Calling `signOut()` removes all tokens from sessionStorage and triggers
   `logoutRedirect()`.

---

## Invariants

- `MsalProvider` is initialized once at the root layout — never per-route.
- `getAccessToken()` never returns an expired token — it refreshes silently or throws.
- The `/login` and `/auth/callback` routes are always publicly accessible.
- Token is never stored in localStorage or a cookie — only in MSAL's sessionStorage cache.
- No business logic lives in `msal-config.ts` or `msal-provider.tsx` — only wiring.

## Edge Cases

| Scenario | Expected behavior |
|----------|-------------------|
| User reloads page mid-session | MSAL restores token from sessionStorage, no re-login needed |
| Refresh token expired | Silent acquisition fails → redirect to `/login` |
| User closes tab | sessionStorage cleared, next visit requires re-login |
| Azure AD returns error in callback | `/auth/callback` reads error param, shows error message, offers retry |
| Popup blocked by browser | Not applicable — using redirect flow, not popup |
| `NEXT_PUBLIC_MSAL_CLIENT_ID` not set | App fails to initialize; console error at startup |
| User navigates directly to `/dashboard` without auth | middleware.ts redirects to `/login` |
| MSAL `handleRedirectPromise()` called outside callback page | Returns `null` silently — no side effects |
| Multiple tabs open | Each tab has its own sessionStorage; independent sessions |
